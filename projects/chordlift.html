ChordLift

I scoped, built, and shipped a full-stack AI guitar practice app, not a slide deck. ChordLift is proof that I am a product manager who uses AI to deliver end-to-end products: problem framing, MVP trade-offs, ML integration, and a live URL at chordlift.pages.dev.

Try live demo

Problem
Guitarists learning from recordings waste time pausing, rewinding, and guessing chords from unreliable tabs.
My role
Product manager and sole builder: problem framing, MVP scope, UX, React frontend, FastAPI backend, ML integration, CI, and deployment. Used AI coding assistants to accelerate implementation while I owned product decisions and architecture.
Toolkit
React, TypeScript, FastAPI, Python, lv-chordia, Docker, GitHub Actions, Cloudflare Pages, Cursor.

Problem statement

Learning guitar from recordings means constant pause-and-rewind loops and chord charts that rarely match the recording you are actually hearing. I saw a product opportunity: turn passive listening into active practice with time-synced chords and neck visualization, then decided to build it myself instead of stopping at a spec.

Situation & task

Situation: I play guitar and felt the friction firsthand: tabs are static, YouTube tutorials are slow, and existing tools either stop at chord names or ignore timing entirely. Task: Define an MVP that delivers real practice value (upload → analyze → play along), ship it to a public URL, and keep enough product discipline to measure ML quality over time, not just ship and hope.

Building with AI as a PM

This was not “vibe-coding a toy.” I treated AI assistants (Cursor and similar tools) as a force multiplier while keeping product ownership on my side of the keyboard.

  • Scoped vertically first. One user journey (upload audio, run analysis, play along with synced chords) before adding YouTube, Spotify, or solo fretboard complexity.
  • AI for speed, PM judgment for direction. Assistants helped with boilerplate, integration glue, and iteration velocity across React, FastAPI, and Docker. I drove architecture, trade-offs, and what actually shipped.
  • Examples of PM calls AI did not make: upload-only production on Cloudflare Pages (VITE_UPLOAD_ONLY) for reliability and legal simplicity; a gold-eval regression pipeline separate from fast unit CI; solo fretboard behind a feature flag until the UX was ready; cached slow analysis vs real-time playback sync; user chord corrections in localStorage instead of chasing perfect ML on v1.
  • Outcome: a deployed product at chordlift.pages.dev, not a Figma mock.

Product decisions & trade-offs

  • Upload-only prod vs multi-source dev. Production accepts file uploads only, which is reliable and legally simple on Cloudflare Pages. Dev builds retain YouTube, Spotify, and SoundCloud ingestion for faster iteration.
  • Cached analysis vs real-time sync. ML chord detection is slow; playback highlighting must be instant. I separated the heavy analysis step from a lightweight sync engine that runs at 60fps during playback.
  • User corrections over perfect ML. Rather than block launch on chord accuracy, I added a correction loop so players can fix mis-detections locally and keep practicing.
  • Feature flags for unfinished surfaces. Scale-practice mode and solo fretboard shipped when ready, not when the codebase happened to compile.

Technical highlights

  • Sync engine. A requestAnimationFrame loop reads audio.currentTime and updates DOM/SVG nodes directly, bypassing React state so chord and note highlighting stays smooth at 60fps.
  • ML pipeline. lv-chordia on an HPSS chord stem with pitch-reliability gating; basic-pitch for solo note detection; rhythm hints estimated from onset density.
  • Scale practice. Standalone mode with pentatonic CAGED shapes, seven diatonic modes, and an interactive SVG fretboard, no upload required.
  • Infra & quality. Docker Compose with Caddy, GitHub Actions CI, and a nightly gold-eval pipeline to catch ML regressions before users do.

Challenges & how I handled them

  • YouTube fragility. Download blockers and ToS risk pushed me toward upload-first production while keeping multi-source ingestion in dev for my own testing.
  • React vs playback performance. Early prototypes stuttered during sync. Moving highlight updates out of React state into direct DOM manipulation fixed it without rewriting the UI layer.
  • ML trust. Chord detection will never be perfect on v1. Shipping corrections and a gold-eval harness let me improve the model without blocking the core practice loop.
  • Solo builder bandwidth. AI assistants compressed full-stack delivery, but I still had to prioritize ruthlessly: one journey, one prod mode, one quality gate at a time.

Results

Live in production chordlift.pages.dev: upload an MP3/WAV and play along with time-synced chords.
